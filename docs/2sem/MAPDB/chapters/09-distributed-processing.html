10  Distributed Processing and MapReduce

10.1 Scaling Processing

As discussed in the previous chapter, processing performance can scale with the number of processing units, provided that the computation is at least partially parallelisable. Speedup studies typically confirm that wall-time scales favourably with \(n\) for heavily data-parallel tasks (SPMD-style).

The natural next step is to tackle very large, possibly distributed, datasets by increasing \(n\) beyond what a single machine can offer. Two fundamental strategies exist:

Strategy Description
Vertical scaling (Scale Up) Increase the capacity or performance of existing hardware — buy a faster machine.
Horizontal scaling (Scale Out) Add more machines — buy more commodity hardware.

Horizontal scaling is almost always preferred for large-scale data processing: commodity hardware is cost-effective, and the system can grow incrementally.

10.2 Parallel Computing Architectures

Different memory-sharing architectures lead to different trade-offs in performance and scalability.

10.2.1 Shared Memory Multiprocessor (SMP)

In a Shared Memory Multiprocessor, a set of CPUs share a common memory space connected by a high-speed bus or fabric.

   CPU      CPU
    |        |
  NETWORK/BUS
    |        |
  SHARED MEMORY
    |        |
   CPU      CPU
  • All modern workstations (including smartphones) are SMP systems.
  • Memory access is uniform and transparent to the programmer.
  • Scaling this architecture to a very large number of processors is expensive and technically demanding.
NUMA: Non-Uniform Memory Access

Multi-socket servers partition memory into NUMA nodes — one per CPU socket, each with local RAM. The lscpu output for a dual-socket server shows NUMA node(s): 2.

Accessing local memory (same NUMA node as the running core) is significantly faster than accessing remote memory (on the other socket). The OS and job schedulers can pin processes to specific NUMA nodes with numactl:

numactl --cpunodebind=0 --membind=0 python3 my_script.py

NUMA topology is a key consideration when scaling shared-memory parallelism beyond a single socket.

10.2.2 Distributed Memory Multiprocessor (Cluster)

In a cluster, each processor has its own local memory — a complete unit called a node. All nodes are connected via a high-speed network.

  MEM.    MEM.
   |       |
  CPU     CPU
    \     /
   NETWORK/BUS
    /     \
  CPU     CPU
   |       |
  MEM.    MEM.
  • Good overall performance even from commodity hardware.
  • The complexity lies in the communication across nodes.
  • Scales horizontally to thousands or millions of nodes.

10.3 Distributed Computing: Clusters

A cluster distributes the processing workload across multiple networked nodes, exposing itself to users as a single virtual machine.

Typical cluster components:

  • Head node — the access point for users; distributes workload to compute nodes.
  • Computing nodes (CN) — where the actual processing runs; usually invisible to users.
  • Interactive nodes — accessible to users as standard computers for testing, storage access, or launching jobs.
  • Specialised nodes — nodes equipped with specific hardware (GPUs, large RAM, fast NVMe) for particular workloads.

10.3.1 Supercomputers as Clusters

Most of today’s supercomputers are clusters:

CINECA LEONARDO (world’s #10 in November 2025) - 1 536 nodes with 2× Intel Sapphire Rapids (56-core CPUs) - 3 456 nodes with 1× Intel Xeon 8358 (32-core CPU) and 4× NVIDIA Ampere GPUs

On the opposite extreme, clusters of Raspberry Pi boards or repurposed desktop PCs (COTS — Commodity Off-The-Shelf hardware) can efficiently tackle a wide range of parallel problems through opportunistic computing (using shared resources when they are idle).

10.4 Batch Systems

Users typically submit jobs to a cluster through a batch system rather than logging in interactively:

  1. A job consists of an executable, optional arguments, and resource requirements (CPUs, memory, walltime, etc.).
  2. The batch scheduler accepts the job and places it in a queue.
  3. The scheduler selects which job runs next based on priority, previous resource usage, and availability.
  4. The job’s tasks are dispatched to computing nodes for execution.

Popular batch schedulers include PBS, SLURM, LSF, and HTCondor. Example HTCondor submission:

$ condor_submit my_job.sub
Submitting job(s).
50 job(s) submitted to cluster 5146936.

$ condor_q
OWNER     BATCH_NAME  SUBMITTED  DONE  RUN  IDLE  TOTAL  JOB_IDS
jpazzini  ID:5146936  10/4 17:29   12   25    13     50   5146936.0-49

An HTCondor submit file:

executable              = my_exec
arguments               = --input input_file_$(ProcId).txt
transfer_input_files    = input_file_$(ProcId).txt
log                     = my_job_$(ProcId).log
error                   = my_job_$(ProcId).err
output                  = my_job_$(ProcId).out
request_disk            = 8GB
request_memory          = 2GB
queue 50

10.5 Programming Frameworks for Parallel-Distributed Systems

Optimising complex tasks for parallel and distributed execution requires knowledge of specific programming standards:

Framework Purpose
OpenMP Parallelisation directives for shared-memory multi-core CPUs on a single node
MPI (Message Passing Interface) Communication and synchronisation across nodes of a cluster
CUDA / HIP Vector/matrix processing on NVIDIA and AMD GPUs

In this course we focus on higher-level, Python-accessible frameworks that abstract away the inner workings of distributed communication — in particular, the MapReduce paradigm and its implementations in Hadoop and Spark.

GPU Computing: Going Further

This course focuses on CPU-based parallelism and distributed processing. GPU programming (CUDA, OpenCL, HIP, Numba, CuPy) and heterogeneous CPU+GPU architectures are covered in the optional companion course “Modern Computing for Physics”, which addresses SIMD/vectorisation, GPU kernel programming, and MPI+GPU hybrid models.

10.6 Big Data and the 3 Vs

From the early 2000s, a wave of data-processing innovations emerged, driven largely by technology companies that needed to handle datasets at an unprecedented scale. These are often summarised as Big Data, characterised by the three Vs:

  • Volume — datasets at the scale of terabytes, petabytes, or beyond.
  • Velocity — data arriving at high speed (streaming sensors, transactions, logs).
  • Variety — structured, semi-structured, and unstructured data.

The five key ingredients of a BigData processing stack are:

  1. APIs / programming language — the interface developers use.
  2. Software framework for distributed processing.
  3. Cluster orchestrator / scheduler — resource management.
  4. Distributed file system (DFS) — storage layer.
  5. Computing nodes — physical or cloud hardware.

10.7 Hadoop

Hadoop was born in 2006 within Yahoo! and was based on a 2003 Google paper on the Google Distributed File System. It rapidly became the de facto standard for BigData storage and processing.

Large-scale real-world deployments have included:

  • Facebook: 1 100-node cluster (8 800 cores, ~12 PB storage); 300-node cluster (~3 PB)
  • Twitter: clusters storing over 500 PB, largest with >10 000 nodes
  • Yahoo!: >100 000 CPUs across >40 000 computers

Hadoop is open source (Apache Foundation), written in Java, and designed to scale horizontally. Its main components are:

Component Role
Hadoop Common Java-based shared libraries and utilities
HDFS Hadoop Distributed File System — storage
YARN Yet Another Resource Negotiator — cluster manager and scheduler
Hadoop MapReduce Distributed programming model

10.8 HDFS: A Reminder

HDFS stores data as large blocks distributed across data nodes, with the name node keeping metadata:

                  Metadata (names, blocks, replicas, …)
                 ┌─────────────────────────────────────┐
Client ──────────► Name Node         Secondary Name Node
                 │                   (checkpoints, backup)
                 │  Block write
                 ▼
          Data Node   Data Node   Data Node   Data Node
               └──── Replication ─────┘
  • The Name Node manages the filesystem namespace and metadata.
  • The Secondary Name Node periodically checkpoints the metadata.
  • Data Nodes store the actual data blocks and replicate them for fault tolerance.

10.9 The MapReduce Paradigm

MapReduce is not a programming language, but a paradigm for developing distributed algorithms on potentially very large datasets. It decomposes computation into two phases:

10.9.1 1. Map

Split the input data into <key, value> pairs, and apply a user-defined map function to each pair, producing a list of intermediate <key, value> pairs:

\[ \text{map}(k_\text{in}, v_\text{in}) \rightarrow \text{list}[(k_1, v_1),\, (k_2, v_2),\, \ldots] \]

Each key-value pair is processed independently, enabling full data parallelism across the cluster.

10.9.2 2. Reduce

Collect all intermediate pairs with the same key, then apply a user-defined aggregation function to produce a final result:

\[ \text{reduce}(\text{list}[(k, v_1), (k, v_2), \ldots]) \rightarrow [(k, v_\text{out})] \]

10.9.3 Word Count Example

The canonical example: count the occurrences of each word in a text corpus.

Input: "Car Car River Deer Car Bear"

Map phase:
  ("Car", 1), ("Car", 1), ("River", 1), ("Deer", 1), ("Car", 1), ("Bear", 1)

Shuffle & Sort (grouping by key):
  "Bear" → [1]
  "Car"  → [1, 1, 1]
  "Deer" → [1]
  "River"→ [1]

Reduce phase (sum):
  Bear  → 1
  Car   → 3
  Deer  → 1
  River → 1

10.10 MapReduce in Hadoop

10.10.1 Map Phase

Data is read from HDFS, partitioned among computing nodes, and sent to each node as a set of key-value pairs. The Map tasks process input records independently and write intermediate results to the local disk of the node. This data locality principle — bringing computation to the data rather than the data to the computation — is central to Hadoop’s performance.

10.10.2 Reduce Phase

Once all Map tasks complete, a shuffle and sort step groups and transfers the intermediate key-value pairs to the Reduce tasks. The Reduce tasks then apply the aggregation function to each group.

10.11 YARN: Yet Another Resource Negotiator

YARN acts as the “operating system” of the Hadoop cluster, managing the scheduling and execution of jobs.

Key components:

Component Role
Resource Manager (RM) Manages cluster-wide resource allocation; contains the Scheduler and the Application Manager
Application Master (AM) Negotiates resources from the RM and coordinates the execution of a single application
Node Manager (NM) Manages the user processes on each individual machine
Container A set of allocated resources (memory, CPU) used to run a specific task

10.12 Hadoop Higher-Level Abstractions

Writing raw MapReduce jobs in Java is verbose. Higher-level tools provide more accessible interfaces:

Tool Description
Pig High-level scripting language translated into MapReduce programs (Yahoo!, latest release 2017)
Hive SQL-like query language (HiveQL) for data stored in HDFS (Facebook/Netflix)
Oozie Workflow scheduler for chaining multiple MapReduce jobs

10.13 The Hadoop Ecosystem

Hadoop grew into a rich ecosystem covering the full BigData stack:

  • APIs / programming language: Pig, Hive, custom Java/Python
  • Distributed processing framework: Hadoop MapReduce
  • Cluster orchestrator: YARN
  • Distributed file system: HDFS

The broader BigData landscape includes many additional tools (Spark, Kafka, Cassandra, HBase, etc.) that either complement or replace individual components of the Hadoop stack depending on the use case.

The full BigData stack can be visualised as five layers, with Hadoop occupying all four lower layers and higher-level tools sitting on top:

┌────────────────────────────────────────────────────────────────┐
│  APIs / Languages    │ Pig · Hive · HiveQL · custom Java/Python │
├──────────────────────┼─────────────────────────────────────────┤
│  Processing          │ Hadoop MapReduce · Apache Spark · Storm  │
├──────────────────────┼─────────────────────────────────────────┤
│  Cluster Scheduler   │ YARN · Mesos · Kubernetes                │
├──────────────────────┼─────────────────────────────────────────┤
│  Distributed FS      │ HDFS · Amazon S3 · Azure Data Lake       │
├──────────────────────┼─────────────────────────────────────────┤
│  Computing Hardware  │ Commodity nodes · Cloud VMs · Bare metal │
└────────────────────────────────────────────────────────────────┘

Beyond this core stack, the broader ecosystem includes: Kafka (streaming), HBase / Cassandra (wide-column NoSQL), ZooKeeper (distributed coordination), Sqoop (RDBMS ↔︎ HDFS transfer), Oozie (workflow scheduling), and Mahout (distributed machine learning).
