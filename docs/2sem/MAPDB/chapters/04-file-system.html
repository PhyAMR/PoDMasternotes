4  File System

There are three main solutions for storing and working with datasets:

  1. File: the best known and most used. Files are sets of data blocks managed by a system called a File System, which distributed over multiple memory units (horizontal scalability) will act as a Distributed File System.
  2. DataBase: each single element is accessible at a granular level and modifiable through systems called DataBase Management System; if distributed over multiple memory units, DataBases are called Distributed DataBases.
  3. ObjectStore: the data are saved in any arbitrary form (files, blocks of specific length, entire operating systems) inside containers called buckets; they are mainly used in cloud storage.

This chapter deals with file systems; the next chapter covers databases.

4.1 Characteristics

The role of a file system is to manage and keep track of the position of data saved in memory in the form of files through a hierarchical structure, making use of folders (also called directories) and subfolders. Its main characteristics can be summarized in three aspects:

  1. Data saving: the data are physically stored in the form of blocks of fixed length, distributed within the memory devices. The file system both divides and allocates the files in memory, and keeps track of all the positions of the blocks.
  2. Abstraction: the file system identifies the data through the name of the file, the path organized according to a hierarchical structure, and the metadata attributed to the file.
  3. Low-level operations: the file system interfaces with the operating system to allow low-level operations on the storage blocks, such as opening, closing, reading, writing, and searching. This interface follows precise standards, the most widespread of which is POSIX (Portable Operating System Interface for Unix).

The POSIX (Portable Operating System Interface) standard defines the minimum set of file operations that a conforming OS must provide:

Operation Description
OPEN Open or create a file; returns a file descriptor
CLOSE Release the file descriptor
READ Read bytes from the current position
WRITE Write bytes at the current position
SEEK Move the current position within the file

POSIX compliance ensures that programs written for one Unix-like OS (Linux, macOS, BSD) will work on another without modification.

4.2 File Access Modes

File systems support three fundamental access patterns:

Access Mode Description Use Case
Sequential Records are read/written in order from beginning to end Log files, streaming data, tape archives
Direct (Random) Any record can be accessed directly by its position/address Databases, indexed data
Indexed An index structure maps keys to physical locations; lookups use the index B-tree indexes in databases, directory listings

Most operating systems implement direct access at the OS level; sequential and indexed access are typically layered on top by applications or database engines.

4.3 Space Allocation

The main distinction between the various file systems concerns the way in which space is allocated to save files in memory. Below are three types of allocation, proceeding by scale of efficiency.

4.3.1 Contiguous Allocation

The various blocks that make up the individual files are physically arranged next to each other. Access to the files is of a sequential type.

The limit of this type of allocation is evident: when you want to expand the content of a file, beyond a certain point you will encounter an already occupied block from another file that cannot be overwritten. For this reason, a file system based on this allocation is suitable for magnetic tapes, where files are saved to be kept long-term without ever being modified.

File Start Length
count 0 2
f 6 2
tr 14 3
mail 19 6
list 28 4

4.3.2 Linked Allocation

Through linked allocation it is possible to “jump” from one block to another to store a single file. This is done by inserting in each block the information on where the next data block is located, so a small part of the memory is “sacrificed” to allow the file system to move and retrieve information.

Compared to contiguous allocation, a better optimization of the available space is possible, not being bound to a series of contiguous blocks to compose the file.

File Start End
jeep 9 25

Example directory table for linked allocation

  • Advantage: no external fragmentation; files can grow dynamically without needing contiguous free space.
  • Disadvantage: random access is O(n) — to reach block k you must follow k pointers sequentially. Additionally, a single corrupt pointer loses the rest of the file.
FAT: Linked Allocation with an In-Memory Table

FAT (File Allocation Table) is a variant of linked allocation where the linked list of block pointers is stored in a separate table kept in memory rather than within the data blocks themselves. This avoids the per-block pointer overhead and allows faster traversal, but the FAT table must fit in memory and becomes a single point of failure.

4.3.3 Indexed Allocation

Linked allocation optimizes space on the disk but not the time in operations involving the files: every time an operation is requested, the system must start from the initial block and “bounce” between all subsequent blocks until it reaches the desired destination.

Indexed allocation optimizes time through the use of direct and indirect blocks, with the latter having the function of indicating the positions of the data.

The first block is constituted as follows:

  • The initial part contains the metadata relating to the file.
  • Then follow the direct blocks — addresses pointing directly to the data.
  • Optionally, single indirect blocks point to blocks that in turn point to the data; double indirect blocks extend this one level further, and so on.

An inode’s data block pointers are organized in tiers:

  1. Direct blocks (e.g., 12 pointers) — each pointer points directly to a data block.
  2. Single indirect block — one pointer to a block that contains N pointers, each pointing to a data block.
  3. Double indirect block — one pointer to a block of pointers, each of which points to a block of pointers to data blocks.
  4. Triple indirect block — one further level of indirection beyond double indirect.

This tiered structure keeps the inode small while supporting very large files via indirection: small files use only direct blocks (fast access), while large files extend through indirect blocks as needed.

Example (Berkeley Fast File System): 12 direct blocks + 1 indirect block, each \(4\ \text{kB}\); memory addresses occupy \(4\) bytes.

The 12 direct blocks contain:

\[12 \times 4\ \text{kB} = 48\ \text{kB}\]

The indirect block contains \(\frac{4\ \text{kB}}{4\ \text{B}} = 1000\) addresses, pointing to \(1000 \times 4\ \text{kB} = 4\ \text{MB}\) of data. Total:

\[48\ \text{kB} + 4\ \text{MB} \approx 4\ \text{MB}\]

Adding a double indirect block would yield \(1000 \times 1000 \times 4\ \text{kB} = 4\ \text{GB}\) more. Increasing the layer of indirect blocks exponentially increases the memory dedicated to a file.

4.4 Limits

A file system is mainly limited by the space it can dedicate to a single file, and by the total space it can manage for a single memory unit (due to counters with a limited number of bits).

Another limitation is that the file system cannot give information on the content within a single file — it is up to the user to exploit the hierarchical tree organization to order the data conveniently. This does not allow operations such as filtering based on the content of the files.

Pros Cons
Can handle a large amount of data Scaling a growing amount of data according to the hierarchical view is complex
Low-level operations via POSIX are very fast No structure underlying that of the files is accessible
Offers a hierarchical view of all data It is not possible to group, select and filter data based on file content
Single entities (files) can be easily transferred, copied, shared and processed

4.5 Common File Systems Compared

File System OS Max File Size Max Volume Size Max Filename Permissions Journal
FAT32 Windows/cross-platform 4 GB 8 TB 255 UCS-2 characters No No
exFAT Windows/cross-platform 128 PB 128 PB 255 UTF-16 characters No No
NTFS Windows 16 TB 256 TB 255 characters Yes Yes
HFS+ macOS 8 EB 8 EB 255 UTF-16 characters Yes Yes
ext3 Linux 2 TB 32 TB 255 bytes Yes Yes
ext4 Linux 16 TB 1 EB 255 bytes Yes Yes
XFS Linux 8 EB 8 EB 255 bytes Yes Yes

ext4 is the default file system on most modern Linux distributions and is suitable for the majority of workloads. FAT32/exFAT are used for cross-platform compatibility (USB drives, SD cards).

File System Paths in Practice

A file path represents the full location of a file within the hierarchy. In the context of physics data, paths can encode experiment, run, and file type information:

/eos/cms/store/data/Run2012D/SingleMu/RAW-RECO/22Jan2013-v1/20000/
    00259D38-8A6F-E211-8E23-003048F118C4.root

Each component — experiment (cms), data type (data), run period (Run2012D), stream (SingleMu), format (RAW-RECO) — is a directory level, making the hierarchy self-documenting.
