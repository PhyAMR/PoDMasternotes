9  Processing and Concurrency

9.1 Introduction to Data Processing

Processing refers broadly to the set of operations performed on a dataset to extract higher-level information. In a very simplified view:

\[ \text{DATA(SET)} \xrightarrow{\text{PROCESSING}} \text{RESULTS} \]

This encompasses a wide variety of tasks such as analytics, data reduction, data cleansing, and feature enrichment. In practice, processing often involves multiple sequential stages, each consuming the output of the previous one:

\[ \text{DATA} \xrightarrow{\text{PROCESSING}} \text{DATA} \xrightarrow{\text{PROCESSING}} \text{DATA} \xrightarrow{\text{PROCESSING}} \text{RESULTS} \]

The fundamental question that runs through all of these stages is: how does the processing scale with the data size, task complexity, and available resources?

9.2 CPU Processing

At its core, processing is the sequential execution of instructions by the CPU, operating on data held in memory. The CPU follows the classic fetch–decode–execute cycle for each instruction:

  • Fetch — retrieve the instruction from memory
  • Decode — interpret the instruction
  • Execute — carry out the operation (Von Neumann architecture)

9.2.1 Processing Time

The CPU processing time is the time needed to execute the entire instruction set. Two complementary metrics matter:

  • Wall-time (response time / elapsed time): the total latency to complete a task, including all I/O operations (memory and storage access), network communication, and OS overhead.
  • CPU time: the time attributed solely to instruction processing by the CPU, excluding time spent waiting for other activities.

CPU time can be modelled as:

\[ t_{\text{CPU}} = \sum_i \frac{IC_i \cdot CPI_i}{CF} \]

where:

Symbol Meaning
\(IC_i\) Instruction count — number of instructions in the program
\(CPI_i\) Clock cycles per instruction
\(CF\) Clock frequency (Hz)

9.2.2 Historical Context: Moore’s Law

In 1965, Gordon Moore observed that the number of transistors on a chip doubled approximately every 18–24 months — a trend that held for nearly five decades and became known as Moore’s Law.

Era Key trend
1970s–1990s Transistor count and clock frequency both doubled regularly
Early 2000s Clock frequency plateau: thermal limits prevented further frequency scaling
2004 onwards Industry shifted to multi-core CPUs rather than faster single cores
Today Transistor scaling continues (though slowing) but single-thread performance improvements are incremental

The plateau in single-thread performance — roughly 3–4 GHz since the mid-2000s — is why parallel and distributed computing became the dominant paradigm for improving application throughput.

9.2.3 The Power Wall

A straightforward way to increase processing performance is to raise the clock frequency. This strategy drove chip performance for many years, alongside transistor scaling.

However, from the early 2000s onwards, processors started hitting a fundamental limit: power dissipation. The power consumed by a chip scales as:

\[ P \propto n \cdot C \cdot V^2 \cdot f \]

where \(n\) is the number of transistors, \(C\) is transistor capacitance, \(V\) is the supply voltage, and \(f\) is the clock frequency. Early 32-bit processors such as the Intel 80386 consumed around 2 W; a 2021 4.0 GHz Intel Xeon Gold 5318Y consumes more than 150 W. Continuing to scale frequency without also reducing voltage became thermally and energetically unsustainable.

9.2.4 Pipelining and Alternative Strategies

As performance gains from raw clock-frequency scaling became harder to sustain, chip designers turned to instruction pipelining, which allows multiple instructions to overlap in their fetch–decode–execute stages on the same ALU. This achieves a speedup without raising the clock frequency.

A broader change of paradigm was therefore required to continue improving performance.

9.3 Multi-Processing Architectures

The industry response to the power wall was multi-core CPUs: placing multiple independent processing units (cores) on the same die, and further combining multiple CPUs (sockets) on the same motherboard.

This can be realised in several ways:

  • Symmetric Multiprocessing (SMP) — homogeneous CPUs, all with equal access to shared memory.
  • Asymmetric Multiprocessing — heterogeneous CPUs with different roles (e.g. big.LITTLE mobile architectures).
  • Master/Slave — one master CPU assigns tasks to subordinate CPUs.

A modern laptop (e.g. a Lenovo X1 Gen 8 with an Intel Core i7-10510U) might show the following lscpu output:

Thread(s) per core:  2          ← Intel Hyper-Threading: 2 independent "feeding lanes" per core
Core(s) per socket:  4          ← 4 physical processing units (cores) per CPU
Socket(s):           1          ← 1 CPU socket on the motherboard

L1d cache:           128 KiB
L2 cache:            1 MiB
L3 cache:            8 MiB
Hyper-Threading

Intel’s Hyper-Threading technology provides two independent fetch/decode lanes per physical core, but only one execution unit (ALU). The OS sees 8 logical CPUs on a 4-core processor, but at most 4 instructions can execute truly in parallel at any moment.

The total number of logical CPUs visible to the OS is:

\[ \text{CPUs} = \text{Sockets} \times \text{Cores per Socket} \times \text{Threads per Core} \]

A production-grade server (e.g. a Dell PowerEdge R750 with two Intel Xeon Gold 5318Y) might expose 96 logical CPUs to the OS (2 sockets × 24 cores × 2 threads). Multi-processor machines are now the standard — the question is how to take advantage of them.

9.4 Parallelism

Parallelism means exploiting multi-processing architectures to solve problems faster. Two orthogonal forms of parallelism exist:

  1. Data-Level Parallelism (DLP) — many data items can be operated on simultaneously. The same operation is applied to all elements in parallel.
  2. Task-Level Parallelism (TLP) — multiple sub-tasks of a process operate independently and in parallel.

In practice, the most efficient solutions combine both.

Parallel processing is the simultaneous execution of a single job split into \(n\) sub-tasks across \(m\) CPUs/cores. The primary goal is to reduce the total execution time (speedup).

9.5 Flynn’s Taxonomy

Flynn’s taxonomy (1966) classifies computer architectures by the number of simultaneous instruction streams and data streams:

Single Data Multiple Data
Single Instruction SISD SIMD
Multiple Instruction MISD MIMD
  • SISD — one instruction on one data stream at a time. No parallelism; e.g. a single-threaded process on a single-core CPU.
  • SIMD — one instruction performed on multiple data streams simultaneously. Enables data-level parallelism. Found in modern CPU AVX/SSE/MMX instructions and most GPUs.
  • MISD — multiple instructions on a single data stream. Rare in practice; used in fault-tolerant applications.
  • MIMD — multiple instructions on multiple data streams simultaneously. Supports both DLP and TLP; each processing unit runs independently. Most modern parallel computers fall into this category.
SPMD

A widely used sub-category of MIMD is SPMD (Single Program Multiple Data): the same entire program (not just single instructions) runs on multiple “chunks” of data across several processing units. This is the natural model for embarrassingly parallel problems.

9.6 Speedup: Amdahl’s Law

Is it possible to speed up a computation indefinitely by adding more processing units?

Given a process with a fraction \(p\) that can be parallelised and a fraction \((1-p)\) that must remain serial, and with \(n\) processing units, the speedup predicted by Amdahl’s law is:

\[ S(n) = \frac{1}{(1 - p) + \dfrac{p}{n}} \]

As \(n \to \infty\):

\[ S_{\max} = \frac{1}{1 - p} \]

No matter how many processing units are available, the serial fraction of the computation remains the bottleneck. To maximise performance, the process itself must be redesigned to reduce its serial portion.

Amdahl’s law is often considered somewhat pessimistic, as it assumes the serial fraction grows proportionally with problem size.

9.7 Gustafson’s Law

Gustafson’s scaled speedup addresses parallelism from a different angle. It observes that in practice the “problem size” often scales with the number of available processors: as more processors are added, a larger problem is tackled, so the parallelisable component grows more than the serial component.

\[ S_{\text{scaled}}(n) = 1 + (n - 1) \cdot p \]

Either way, heterogeneous computing — integrating CPUs, GPUs, and specialised accelerators — aims to maximise both the serial and parallel components of a computation.

9.8 Processes and Threads

9.8.1 Definitions

Process

A process is an instance of a program in execution. Every process is an independent entity with its own executable code and memory space. Multiple instances of the same program can run simultaneously — they share the same code but are distinct processes.

Thread

A thread is the basic unit of CPU utilisation. Every thread consists of a program counter, a set of registers, and its own execution stack. A thread is part of a process, and each process can contain one or more threads.

Within a multi-threaded process:

  • All threads share the process memory and common resources (“global” for all threads).
  • Each thread has an independent set of registers, memory, and execution state.

A word processor, for example, might use one thread to check spelling, another to process keystrokes, a third to load images, and a fourth to perform periodic automatic saves.

9.8.2 Concurrency vs Parallelism

These two concepts are often confused:

  • Concurrency — multiple tasks are in progress at the same time. Tasks can start, run, and stop independently and in any order. On a single-core CPU, concurrency is achieved through rapid context switching.
  • Parallelism — multiple tasks are executing at the same time. True simultaneous execution, requiring multiple processing units.

9.8.3 Multi-Tasking

Multi-tasking is the OS-enabled execution of multiple concurrent tasks. The OS instructs the CPU to share its time between tasks via a scheduler:

  • Pre-emptive scheduling — the scheduler alternates between \(N\) tasks, sharing CPU time approximately equally (with priority adjustments).
  • Collaborative scheduling — the system switches processes only when the active one voluntarily releases control.

A context switch saves the state of the currently running process or thread so it can be resumed later.

9.8.4 Multi-Threading

Multi-threading is the execution of multiple threads within a process. The execution mode depends on the architecture:

  • Concurrent, on single-core CPUs with a multi-tasking OS (the CPU alternates between threads via context switching).
  • Parallel, on multi-core CPUs (different threads run simultaneously on different cores).

Context switching between threads is lighter than between full processes, since threads within the same process share memory. Multiple sub-components of a single application can thus proceed independently while sharing global resources.

9.9 Race Conditions

Multi-threaded execution can lead to subtle bugs when multiple threads access the same shared memory simultaneously.

Race Condition

A race condition occurs when a shared resource (e.g. a variable in memory) is accessed concurrently by multiple threads, and the outcome depends on the particular order in which the accesses take place.

Consider var_x = 5 shared between two threads:

  • Thread 1 reads var_x (value 5) and computes 5 * 2 = 10.
  • Thread 2 also reads var_x (still 5, before Thread 1 has written back) and computes 5 * 3 = 15.
  • Thread 1 writes 10 back to var_x.
  • Thread 2 writes 15 back to var_x.
  • Final value: 15 — but the correct sequential result (multiply by 2 then by 3) should be 30.

9.10 Thread Synchronisation

Thread synchronisation mechanisms prevent race conditions by controlling access to shared resources:

  • Lock / Mutex (Mutual Exclusion) — only one thread at a time can hold the lock and access the protected resource. Other threads block until the lock is released.
  • Binary semaphore — similar to a mutex, with a 0/1 counter.
  • Counting semaphore — controls access to a pool of resources.
Deadlock

A deadlock occurs when a thread waits indefinitely for an event that can only be triggered by another thread that is itself blocked. Locks must always be released after use, and circular wait conditions must be avoided.

9.11 The Python GIL

The Global Interpreter Lock (GIL) is a mutex on the CPython interpreter itself, ensuring that only one thread at a time can hold the GIL and thus execute Python bytecode. This design:

  • Prevents many deadlocks and race conditions at the interpreter level.
  • Disables true parallelism for CPU-bound work in CPython: even on a multi-core machine, two Python threads from the same process cannot execute Python bytecode simultaneously.

The GIL is an implementation detail of CPython (the standard Python interpreter), not a requirement of the Python language. Alternative interpreters such as Jython and IronPython do not rely on it.

Python 3.13 and the GIL

Starting with Python 3.13 (October 2024), the GIL became optional. CPython now provides a “free-threaded” build mode in which the GIL is disabled, enabling full multi-threading for CPU-bound Python code. The standard build still includes the GIL for backwards compatibility.

GIL Does Not Eliminate All Race Conditions

Even with the GIL, race conditions can occur during OS-level context switches. The GIL is released periodically — by default every 5 ms (sys.getswitchinterval()) — and briefly between the old thread releasing it and the new thread acquiring it. During that window, the OS scheduler may preempt execution mid-operation.

Compound operations that appear atomic in Python source code (e.g. x += 1, which compiles to LOAD/ADD/STORE bytecode instructions) are not guaranteed to be atomic under concurrent thread execution. A lock (threading.Lock) must still be used to protect shared state.

For a detailed analysis of the GIL’s interaction with the OS scheduler, see: https://www.dabeaz.com/python/GIL.pdf

9.12 Threading vs Multiprocessing in Python

Threading (threading) Multiprocessing (multiprocessing)
Memory Shared across threads Separate address space per process
GIL Constrained by GIL (one thread executes at a time) No GIL constraint (each process has its own interpreter)
Best for I/O-bound tasks (disk, network) CPU-bound tasks (numerical computation)
Communication Shared variables (but need locks) Queues, pipes, shared memory objects

Other libraries extend these capabilities: joblib provides a unified interface to both threading and multiprocessing backends.

# joblib example: run square() in parallel across 4 processes
from joblib import Parallel, delayed

def square(x):
    return x ** 2

results = Parallel(n_jobs=4, backend='multiprocessing')(
    delayed(square)(i) for i in range(10)
)
# results = [0, 1, 4, 9, 16, 25, 36, 49, 64, 81]

mpi4py exposes the MPI (Message Passing Interface) standard to Python, enabling distributed-memory parallelism across cluster nodes — the same model used in large-scale HPC applications.

9.13 Hands-on: Concurrency in Python

The following scripts demonstrate the progression from single-threaded execution through multi-threaded and multi-process programming in Python.

Note: These scripts are designed to be run from the terminal (python3 script_name.py). They are not executed during book rendering.

9.13.1 00 — Single-Threaded Baseline

A simple CPU-bound task: sum all integers in a range. This serves as the performance baseline before introducing concurrency. The wall-time here is the cost of pure sequential execution on a single core.

# 00_single_threaded.py
import time

def my_sum(a, b):
    """A simple function summing integers from a to b."""
    the_sum = 0    
    for i in range(a, b):
        the_sum += i
    print(f'Sum from {a} to {b} = {the_sum}')

if __name__ == '__main__':
    # Common "global" objects
    MIN = 0
    MAX = 100_000_000

    # Start a timer
    start = time.time()

    # Run the sum function
    my_sum(MIN, MAX)

    # Stop the timer
    end = time.time()

    print()
    print(f'Time taken = {end - start:.2f} sec')

9.13.2 01 — Two Concurrent Threads

The same summation split across two threading.Thread instances. start() is non-blocking (launches the thread), while join() blocks until that thread finishes. Due to the GIL, wall-time will not improve for CPU-bound work — both threads still share a single interpreter lock.

# 01_multi_threaded.py
import time
from threading import Thread

def my_sum(a, b):
    """A simple function summing integers from a to b."""
    the_sum = 0    
    for i in range(a, b):
        the_sum += i
    print(f'Sum from {a} to {b} = {the_sum}')

if __name__ == '__main__':
    # Common "global" objects
    MIN = 0
    MAX = 100_000_000

    # Create 2 threads
    #
    # Both acting on the same function
    # Each summing half of the values
    t1 = Thread(target=my_sum, args=(MIN, MAX//2,))
    t2 = Thread(target=my_sum, args=(MAX//2, MAX,))

    # Start a timer
    start = time.time()

    # Start both threads 
    #
    # Thread #1 and Thread #2 are started ~ at the same time 
    # and will be executed concurrently or in parallel (depending on the architecture)
    # 
    # `thread.start()` is a **non-blocking** operation 
    # i.e. the Python interpreter creates and starts the threads 
    # and proceeds onward the execution of this script without blocking its execution
    t1.start()
    t2.start()

    # Join both threads 
    #
    # `thread.join()` is a **blocking** call 
    # i.e. it waits for a thread to have completed its task before proceeding onward
    #
    # ??? What would happen if we removed the join call and execute the script ???
    t1.join()
    t2.join()

    # Stop the timer
    end = time.time()

    print()
    print(f'Time taken = {end - start:.2f} sec')

9.13.3 02 — Thread-Local Data

Uses threading.local() to create storage that is private to each thread. Even though the variable name is the same across threads, each thread sees its own independent copy — no sharing, no synchronisation needed.

# 02_multi_threaded_local_data.py
import time
import threading
from threading import Thread

def my_sum(a, b):
    """A simple function summing integers from a to b."""
    
    # Create an instance of local (per-thread) memory 
    # Now "the_sum" is **explicitly** thread-specific
    the_sum = threading.local() 

    the_sum = 0    
    for i in range(a, b):
        the_sum += i
    print(f'Sum from {a} to {b} = {the_sum}')

if __name__ == '__main__':
    # Common "global" objects
    MIN = 0
    MAX = 100_000_000

    # Create 2 threads
    t1 = Thread(target=my_sum, args=(MIN, MAX//2,))
    t2 = Thread(target=my_sum, args=(MAX//2, MAX,))

    # Start a timer
    start = time.time()

    # Start both threads 
    t1.start()
    t2.start()

    # Join both threads 
    t1.join()
    t2.join()

    # Stop the timer
    end = time.time()

    print()
    print(f'Time taken = {end - start:.2f} sec')

9.13.4 03extra — Minimal Race Condition

Ten threads each increment a shared counter 1 000 000 times without any lock. The read–modify–write sequence (counter += 1) is not atomic, so threads interleave and overwrite each other’s updates. The final value is almost certainly less than 10 000 000.

# 03extra_minimal_race_condition.py
import threading

# Increment function using a common counter
def increment(max_val=1_000_000):
    global counter
    for _ in range(max_val):
        counter += 1
    
if __name__ == '__main__':
    # Common counter
    counter = 0

    # Create and start 10 concurrent threads
    threads = []
    for i in range(10):
        t = threading.Thread(target=increment)
        threads.append(t)
        t.start()

    # Join all threads
    for t in threads:
        t.join()
        
    print(f"Final value of counter: {counter}")

9.13.5 03 — Multi-Threaded Without Lock (Race Condition)

Two threads concurrently write to COMMON_COUNTER and COMMON_LIST with no synchronisation. The mismatch between the counter and the list length at the end reveals the race condition in action.

# 03_multi_threaded_nolock.py
import time
import threading
from threading import Thread
from colorama import Fore, Style

def my_sum_mod(a, b, m):
    """A simple function summing integers from a to b.

    Also check whether the current value is a multiple of `m`; if so:
      - add +1 to a common counter `COMMON_COUNTER`
      - store the current value in a common list `COMMON_LIST`
    """

    # Use the `COMMON_COUNTER` and `COMMON_LIST` variables, common to all threads
    global COMMON_COUNTER
    global COMMON_LIST

    the_sum = threading.local() 

    the_sum = 0    
    for i in range(a, b):
        the_sum += i
        if i % m == 0:
            # Modify the common objects
            COMMON_COUNTER += 1
            COMMON_LIST.append(i)

    print(f'Sum from {a} to {b} = {the_sum}')

if __name__ == '__main__':
    # Common "global" objects
    MIN = 0
    MAX = 100_000_000
    MOD = 1397
    COMMON_COUNTER = 0 
    COMMON_LIST    = [] 

    # Create 2 threads
    t1 = Thread(target=my_sum_mod, args=(MIN, MAX//2, MOD))
    t2 = Thread(target=my_sum_mod, args=(MAX//2, MAX, MOD))

    # Start a timer
    start = time.time()

    # Start both threads  
    t1.start()
    t2.start()

    # Join both threads 
    t1.join()
    t2.join()

    # Stop the timer
    end = time.time()

    # Verify if the `COMMON_COUNTER` is equal to the length of `COMMON_LIST`
    if COMMON_COUNTER != len(COMMON_LIST):
        print(Fore.RED + Style.BRIGHT)
    print()
    print(f'Number of multiples of {MOD} in the sum from {MIN} to {MAX} = {COMMON_COUNTER}')
    print()
    print(f'First 100 items of the list of multiples of {MOD}:')
    print(COMMON_LIST[:100])
    print()
    print(f'Cross-check of the size of the list: {len(COMMON_LIST)}')
    print()
    print(f'Time taken = {end - start:.2f} sec')
    print(Style.RESET_ALL)

9.13.6 04extra — Minimal Race Condition Fixed with Lock

The counter example from above, now protected by threading.Lock(). Every counter += 1 is bracketed by acquire() / release(), so only one thread modifies the counter at a time. The final value is always exactly 10 000 000.

# 04extra_minimal_race_condition_with_lock.py
import threading

# Increment function using a common counter
def increment(max_val=1_000_000):
    global counter
    for _ in range(max_val):
        lock.acquire()
        counter += 1
        lock.release()
    
if __name__ == '__main__':
    # Common counter
    counter = 0
    lock = threading.Lock()

    # Create and start 10 concurrent threads
    threads = []
    for i in range(10):
        t = threading.Thread(target=increment)
        threads.append(t)
        t.start()

    # Join all threads
    for t in threads:
        t.join()
        
    print(f"Final value of counter: {counter}")

9.13.7 04 — Multi-Threaded With Lock

The full race-condition scenario corrected: each access to COMMON_COUNTER and COMMON_LIST is guarded by the_lock.acquire() / the_lock.release(). The lock is passed explicitly as an argument so each thread uses the same shared lock object.

# 04_multi_threaded_lock.py
import time
import threading
from threading import Thread
from threading import Lock

def my_sum_mod(the_lock, a, b, m):
    """A simple function summing integers from a to b.

    Also check whether the current value is a multiple of `m`; if so:
      - add +1 to a common counter `COMMON_COUNTER`
      - store the current value in a common list `COMMON_LIST`
    """

    # Use the `COMMON_COUNTER` and `COMMON_LIST` variables, common to all threads
    global COMMON_COUNTER
    global COMMON_LIST

    the_sum = threading.local() 

    the_sum = 0    
    for i in range(a, b):
        the_sum += i
        if i % m == 0:
            # The thread acquires the lock
            the_lock.acquire()
            
            # Code that could result in race condition
            # --------------------- 
            COMMON_COUNTER += 1
            COMMON_LIST.append(i)
            # ---------------------
            
            # The thread releases the lock
            # **Remember to release the lock in order to avoid deadlock conditions**
            the_lock.release()

    print(f'Sum from {a} to {b} = {the_sum}')

if __name__ == '__main__':
    # Common "global" objects
    MIN = 0
    MAX = 100_000_000
    MOD = 1397
    COMMON_COUNTER = 0 
    COMMON_LIST    = [] 

    # Create a thread lock
    LOCK = Lock()

    # Create 2 threads
    t1 = Thread(target=my_sum_mod, args=(LOCK, MIN, MAX//2, MOD))
    t2 = Thread(target=my_sum_mod, args=(LOCK, MAX//2, MAX, MOD))

    # Start a timer
    start = time.time()

    # Start both threads  
    t1.start()
    t2.start()

    # Join both threads 
    t1.join()
    t2.join()

    # Stop the timer
    end = time.time()

    print()
    print(f'Number of multiples of {MOD} in the sum from {MIN} to {MAX} = {COMMON_COUNTER}')
    print()
    print(f'First 100 items of the list of multiples of {MOD}:')
    print(COMMON_LIST[:100])
    print()
    print(f'Cross-check of the size of the list: {len(COMMON_LIST)}')
    print()
    print(f'Time taken = {end - start:.2f} sec')

9.13.8 05 — Thread Pool with Lock (concurrent.futures)

ThreadPoolExecutor manages a pool of reusable worker threads. submit() dispatches a task and returns a Future; wait() blocks until all futures complete. The return value from each thread is retrieved with f.result() — a cleaner pattern than writing to global state.

# 05_multi_threaded_pool_lock.py
import time
from concurrent.futures import ThreadPoolExecutor
from concurrent.futures import wait
import threading
from threading import Lock

def my_sum_mod(the_lock, a, b, m):
    """A simple function summing integers from a to b.

    Also check whether the current value is a multiple of `m`; if so:
      - add +1 to a common counter `COMMON_COUNTER`
      - store the current value in a common list `COMMON_LIST`

Replaces the explicit lock with a queue.Queue, which is inherently thread-safe (its internal lock is managed automatically). Threads call put() to deposit results; the main thread reads them after all workers finish. Queues are the idiomatic Python pattern for producer–consumer communication.

import time

from concurrent.futures import wait

9.13.10 07 — Multi-Processing

multiprocessing.Process spawns a completely separate OS process with its own Python interpreter and memory space, bypassing the GIL entirely. True CPU parallelism is possible: both processes can execute Python bytecode simultaneously on different cores.

# 07_multi_processing.py
import time
from multiprocessing import Process

def my_sum(a, b):
    """A simple function summing integers from a to b."""
    the_sum = 0    
    for i in range(a, b):
        the_sum += i
    print(f'Sum from {a} to {b} = {the_sum}')

if __name__ == '__main__':
    # Common "global" objects
    MIN = 0
    MAX = 100_000_000

    # Create 2 processes (actually... "sub-processes")
    #
    # Both acting on the same function
    # Each summing half of the values
    p1 = Process(target=my_sum, args=(MIN, MAX//2,))
    p2 = Process(target=my_sum, args=(MAX//2, MAX,))

    # Start a timer
    start = time.time()

    # Start both processes
    #
    # Process #1 and Process #2 are started ~ at the same time 
    # and will be executed concurrently or in parallel (depending on the architecture)
    # 
    # `process.start()` is a **non-blocking** operation 
    p1.start()
    p2.start()

    # Join both processes 
    p1.join()
    p2.join()

    # Stop the timer
    end = time.time()

    print()
    print(f'Time taken = {end - start:.2f} sec')

9.13.11 Synchronous vs Asynchronous Execution

Before the next two scripts, it is useful to define these terms precisely:

  • Synchronous (blocking): the calling code waits for the operation to complete before continuing. Execution is sequential.
  • Asynchronous (non-blocking): the calling code submits the operation and continues immediately. Results are collected later via a handle or callback.

In the context of multiprocessing.Pool: - starmap() → synchronous: blocks until all worker processes finish. - starmap_async() → asynchronous: returns an AsyncResult immediately; call .get() when results are needed.

9.13.12 08 — Multiprocessing Pool (starmap, blocking)

Pool.starmap() distributes a list of argument tuples across a pool of worker processes. The call blocks until all workers finish, then returns the collected results as a list. Each worker independently sums its sub-range and returns the partial sum.

# 08_multi_processing_pool_starmap.py
import time
from multiprocessing import Pool

def my_sum(a, b):
    """A simple function summing integers from a to b."""
    the_sum = 0    
    for i in range(a, b):
        the_sum += i
    print(f'Sum from {a} to {b} = {the_sum}')
    return the_sum

if __name__ == '__main__':
    # Common "global" objects
    MIN = 0
    MAX = 100_000_000

    # Define the number of processes
    N_PROCESSES = 4

    # Start a timer
    start = time.time()

    # Create a pool of processes
    pool = Pool(N_PROCESSES)

    # Submit multiple instances of the function `my_sum` 
    # - starmap: allows running the processes with an (iterable) list of arguments
    # - map    : is a similar function, supporting a single argument
    results = pool.starmap(my_sum, [(MIN + _*(MAX - MIN)//N_PROCESSES, MIN + (_+1)*(MAX - MIN)//N_PROCESSES) \
                                    for _ in range(N_PROCESSES)])

    #
    # This means that the main program execution will not 
    # get past them until all sub-processes are completed
    
    # Get the results
    print(f'List of process results: {results}')
    print(f'Total sum = {sum(results)}')

    # Stop the timer
    end = time.time()

    print()
    print(f'Time taken = {end - start:.2f} sec')

9.13.13 09 — Multiprocessing Pool (starmap_async, non-blocking)

starmap_async() returns immediately with an AsyncResult object, allowing the main process to continue (or poll) while workers run. .get() blocks until all results are ready. This pattern is useful when the main process has other work to do while waiting.

# 09_multi_processing_pool_starmapasync.py
import time
from multiprocessing import Pool

def my_sum(a, b):
    """A simple function summing integers from a to b."""
    the_sum = 0    
    for i in range(a, b):
        the_sum += i

    start = time.time()

    pool = Pool(N_PROCESSES)

    # - map_async    : is a similar function, supporting a single argument

        print("Still not ready yet")

    # The `get()` function is therefore **blocking** (equivalent to `join`) 
    results = future_results.get()

    print(f'List of process results: {results}')
    print(f'Total sum = {sum(results)}')

    end = time.time()

    print()
    print(f'Time taken = {end - start:.2f} sec')

9.14.2 CPU-Bound Tasks

CPU-bound tasks are limited by raw computation speed. Because the GIL serialises Python bytecode execution, threading provides no speedup for CPU-bound code — threads take turns rather than running in parallel. Multiprocessing spawns separate interpreter instances that truly run in parallel on different cores.

The example performs a Monte Carlo numerical integration of \(f(x) = \sin^2\!\left(\frac{1}{x(2-x)}\right)\) over \([0, 2]\) across multiple independent chunks.

Single-threaded — 10 chunks processed one after another:

# cpu_intensive.py
import time
import math
import random
from concurrent.futures import ThreadPoolExecutor, ProcessPoolExecutor, wait
from multiprocessing import Pool

def f(x):
    """An arbitrary mathematical function"""
    return (math.sin(1/(x*(2-x))))**2
            
def cpu_function(seed):
    """A function performing an MC integration 
    of `f(x)` taking as input a given seed.
    """
    time_start = time.time()

    # Number of points to generate
    _n = 2_000_000

    # Initialize a local random number generator
    local_random = random
    local_random.seed(seed)

    # Perform the numerical integration
    _count = 0
    for _ in range(_n):
        x = 2 * local_random.random()
        y = local_random.random()
        if y < f(x):
            _count += 1

    print(f'time to perform numerical integration over {_n} entries = {time.time()-time_start:.2f} sec')

    # Return the count
    return (_count, _n)
    
    
if __name__ == '__main__':
    
    # Start a timer
    start = time.time()

    # Number of chunks
    n_chunks = 10

    # Total number of points and count
    total_n = 0
    total_count = 0

    # Split the integral computation in N chunks
    # Use a different number as seed for each random generator
    for _ in range(n_chunks):
        tmp_count, tmp_n = cpu_function(_)
        total_count += tmp_count
        total_n += tmp_n

    # Compute the integral
    integral = 2 * total_count / total_n

    # Stop the timer
    end = time.time()
    
    print()
    print(f'Integral evaluated over {total_n} points = {integral}')
    print()
    print(f'Time taken = {end - start:.2f} sec')

# Processing and Concurrency

## Introduction to Data Processing

**Processing** refers broadly to the set of operations performed on a dataset to extract higher-level information. In a very simplified view:

$$
\text{DATA(SET)} \xrightarrow{\text{PROCESSING}} \text{RESULTS}
$$

This encompasses a wide variety of tasks such as analytics, data reduction, data cleansing, and feature enrichment. In practice, processing often involves multiple sequential stages, each consuming the output of the previous one:

$$
\text{DATA} \xrightarrow{\text{PROCESSING}} \text{DATA} \xrightarrow{\text{PROCESSING}} \text{DATA} \xrightarrow{\text{PROCESSING}} \text{RESULTS}
$$

The fundamental question that runs through all of these stages is: **how does the processing scale with the data size, task complexity, and available resources?**

## CPU Processing

At its core, processing is the sequential execution of instructions by the CPU, operating on data held in memory. The CPU follows the classic **fetch–decode–execute** cycle for each instruction:

- **Fetch** — retrieve the instruction from memory
- **Decode** — interpret the instruction
- **Execute** — carry out the operation (Von Neumann architecture)

### Processing Time

The **CPU processing time** is the time needed to execute the entire instruction set. Two complementary metrics matter:

- **Wall-time** (response time / elapsed time): the total latency to complete a task, including all I/O operations (memory and storage access), network communication, and OS overhead.
- **CPU time**: the time attributed solely to instruction processing by the CPU, excluding time spent waiting for other activities.

CPU time can be modelled as:

$$
t_{\text{CPU}} = \sum_i \frac{IC_i \cdot CPI_i}{CF}
$$

where:

| Symbol | Meaning |
|--------|---------|
| $IC_i$ | Instruction count — number of instructions in the program |
| $CPI_i$ | Clock cycles per instruction |
| $CF$ | Clock frequency (Hz) |

### Historical Context: Moore's Law

In 1965, Gordon Moore observed that the number of transistors on a chip doubled approximately every 18–24 months — a trend that held for nearly five decades and became known as **Moore's Law**.

| Era | Key trend |
|---|---|
| 1970s–1990s | Transistor count and clock frequency both doubled regularly |
| Early 2000s | Clock frequency plateau: thermal limits prevented further frequency scaling |
| 2004 onwards | Industry shifted to multi-core CPUs rather than faster single cores |
| Today | Transistor scaling continues (though slowing) but single-thread performance improvements are incremental |

The plateau in single-thread performance — roughly 3–4 GHz since the mid-2000s — is why parallel and distributed computing became the dominant paradigm for improving application throughput.

### The Power Wall

A straightforward way to increase processing performance is to raise the clock frequency. This strategy drove chip performance for many years, alongside transistor scaling.

However, from the early 2000s onwards, processors started hitting a fundamental limit: **power dissipation**. The power consumed by a chip scales as:

$$
P \propto n \cdot C \cdot V^2 \cdot f
$$

where $n$ is the number of transistors, $C$ is transistor capacitance, $V$ is the supply voltage, and $f$ is the clock frequency. Early 32-bit processors such as the Intel 80386 consumed around 2 W; a 2021 4.0 GHz Intel Xeon Gold 5318Y consumes more than 150 W. Continuing to scale frequency without also reducing voltage became thermally and energetically unsustainable.

### Pipelining and Alternative Strategies

As performance gains from raw clock-frequency scaling became harder to sustain, chip designers turned to **instruction pipelining**, which allows multiple instructions to overlap in their fetch–decode–execute stages on the same ALU. This achieves a speedup without raising the clock frequency.

A broader change of paradigm was therefore required to continue improving performance.

## Multi-Processing Architectures

The industry response to the power wall was **multi-core** CPUs: placing multiple independent processing units (cores) on the same die, and further combining multiple CPUs (sockets) on the same motherboard.

This can be realised in several ways:

- **Symmetric Multiprocessing (SMP)** — homogeneous CPUs, all with equal access to shared memory.
- **Asymmetric Multiprocessing** — heterogeneous CPUs with different roles (e.g. big.LITTLE mobile architectures).
- **Master/Slave** — one master CPU assigns tasks to subordinate CPUs.

A modern laptop (e.g. a Lenovo X1 Gen 8 with an Intel Core i7-10510U) might show the following `lscpu` output:

```

```

::: {.callout-note}
## Hyper-Threading
Intel's Hyper-Threading technology provides two independent **fetch/decode** lanes per physical core, but only one execution unit (ALU). The OS sees 8 logical CPUs on a 4-core processor, but at most 4 instructions can execute truly in parallel at any moment.
:::

The total number of logical CPUs visible to the OS is:

$$
\text{CPUs} = \text{Sockets} \times \text{Cores per Socket} \times \text{Threads per Core}
$$

A production-grade server (e.g. a Dell PowerEdge R750 with two Intel Xeon Gold 5318Y) might expose 96 logical CPUs to the OS (2 sockets × 24 cores × 2 threads). Multi-processor machines are now the standard — the question is how to take advantage of them.

## Parallelism

**Parallelism** means exploiting multi-processing architectures to solve problems faster. Two orthogonal forms of parallelism exist:

1. **Data-Level Parallelism (DLP)** — many data items can be operated on simultaneously. The same operation is applied to all elements in parallel.
2. **Task-Level Parallelism (TLP)** — multiple sub-tasks of a process operate independently and in parallel.

In practice, the most efficient solutions combine both.

**Parallel processing** is the simultaneous execution of a single job split into $n$ sub-tasks across $m$ CPUs/cores. The primary goal is to reduce the total execution time (**speedup**).

## Flynn's Taxonomy

Flynn's taxonomy (1966) classifies computer architectures by the number of simultaneous instruction streams and data streams:

| | Single Data | Multiple Data |
|---|---|---|
| **Single Instruction** | **SISD** | **SIMD** |
| **Multiple Instruction** | **MISD** | **MIMD** |

- **SISD** — one instruction on one data stream at a time. No parallelism; e.g. a single-threaded process on a single-core CPU.
- **SIMD** — one instruction performed on multiple data streams simultaneously. Enables data-level parallelism. Found in modern CPU AVX/SSE/MMX instructions and most GPUs.
- **MISD** — multiple instructions on a single data stream. Rare in practice; used in fault-tolerant applications.
- **MIMD** — multiple instructions on multiple data streams simultaneously. Supports both DLP and TLP; each processing unit runs independently. Most modern parallel computers fall into this category.

::: {.callout-note}
## SPMD
A widely used sub-category of MIMD is **SPMD (Single Program Multiple Data)**: the same entire program (not just single instructions) runs on multiple "chunks" of data across several processing units. This is the natural model for **embarrassingly parallel** problems.
:::

## Speedup: Amdahl's Law

Is it possible to speed up a computation indefinitely by adding more processing units?

Given a process with a fraction $p$ that can be parallelised and a fraction $(1-p)$ that must remain serial, and with $n$ processing units, the speedup predicted by **Amdahl's law** is:

$$
S(n) = \frac{1}{(1 - p) + \dfrac{p}{n}}
$$

As $n \to \infty$:

$$
S_{\max} = \frac{1}{1 - p}
$$

No matter how many processing units are available, the **serial fraction** of the computation remains the bottleneck. To maximise performance, the process itself must be redesigned to reduce its serial portion.

Amdahl's law is often considered somewhat pessimistic, as it assumes the serial fraction grows proportionally with problem size.

## Gustafson's Law

Gustafson's **scaled speedup** addresses parallelism from a different angle. It observes that in practice the "problem size" often scales with the number of available processors: as more processors are added, a larger problem is tackled, so the parallelisable component grows more than the serial component.

$$
S_{\text{scaled}}(n) = 1 + (n - 1) \cdot p
$$

Either way, **heterogeneous computing** — integrating CPUs, GPUs, and specialised accelerators — aims to maximise both the serial and parallel components of a computation.

## Processes and Threads

### Definitions

::: {.callout-note}
## Process
A **process** is an instance of a program in execution. Every process is an independent entity with its own executable code and memory space. Multiple instances of the same program can run simultaneously — they share the same code but are distinct processes.
:::

::: {.callout-note}
## Thread
A **thread** is the basic unit of CPU utilisation. Every thread consists of a program counter, a set of registers, and its own execution stack. A thread is part of a process, and each process can contain one or more threads.
:::

Within a multi-threaded process:

- All threads **share** the process memory and common resources ("global" for all threads).
- Each thread has an **independent** set of registers, memory, and execution state.

A word processor, for example, might use one thread to check spelling, another to process keystrokes, a third to load images, and a fourth to perform periodic automatic saves.

### Concurrency vs Parallelism

These two concepts are often confused:

- **Concurrency** — multiple tasks are *in progress* at the same time. Tasks can start, run, and stop independently and in any order. On a single-core CPU, concurrency is achieved through rapid context switching.
- **Parallelism** — multiple tasks are *executing* at the same time. True simultaneous execution, requiring multiple processing units.

### Multi-Tasking

Multi-tasking is the OS-enabled execution of multiple concurrent tasks. The OS instructs the CPU to share its time between tasks via a **scheduler**:

- **Pre-emptive scheduling** — the scheduler alternates between $N$ tasks, sharing CPU time approximately equally (with priority adjustments).
- **Collaborative scheduling** — the system switches processes only when the active one voluntarily releases control.

A **context switch** saves the state of the currently running process or thread so it can be resumed later.

### Multi-Threading

Multi-threading is the execution of multiple threads within a process. The execution mode depends on the architecture:

- **Concurrent**, on single-core CPUs with a multi-tasking OS (the CPU alternates between threads via context switching).
- **Parallel**, on multi-core CPUs (different threads run simultaneously on different cores).

Context switching between threads is lighter than between full processes, since threads within the same process share memory. Multiple sub-components of a single application can thus proceed independently while sharing global resources.

## Race Conditions

Multi-threaded execution can lead to subtle bugs when multiple threads access the same shared memory simultaneously.

::: {.callout-warning}
## Race Condition
A **race condition** occurs when a shared resource (e.g. a variable in memory) is accessed concurrently by multiple threads, and the outcome depends on the particular order in which the accesses take place.
:::

Consider `var_x = 5` shared between two threads:

- Thread 1 reads `var_x` (value 5) and computes `5 * 2 = 10`.
- Thread 2 **also** reads `var_x` (still 5, before Thread 1 has written back) and computes `5 * 3 = 15`.
- Thread 1 writes `10` back to `var_x`.
- Thread 2 writes `15` back to `var_x`.
- Final value: `15` — but the correct sequential result (multiply by 2 then by 3) should be `30`.

## Thread Synchronisation

Thread synchronisation mechanisms prevent race conditions by controlling access to shared resources:

- **Lock / Mutex (Mutual Exclusion)** — only one thread at a time can hold the lock and access the protected resource. Other threads block until the lock is released.
- **Binary semaphore** — similar to a mutex, with a 0/1 counter.
- **Counting semaphore** — controls access to a pool of resources.

::: {.callout-warning}
## Deadlock
A **deadlock** occurs when a thread waits indefinitely for an event that can only be triggered by another thread that is itself blocked. Locks must always be released after use, and circular wait conditions must be avoided.
:::

## The Python GIL

The **Global Interpreter Lock (GIL)** is a mutex on the CPython interpreter itself, ensuring that only one thread at a time can hold the GIL and thus execute Python bytecode. This design:

- Prevents many deadlocks and race conditions at the interpreter level.
- **Disables true parallelism** for CPU-bound work in CPython: even on a multi-core machine, two Python threads from the same process cannot execute Python bytecode simultaneously.

The GIL is an implementation detail of CPython (the standard Python interpreter), not a requirement of the Python language. Alternative interpreters such as Jython and IronPython do not rely on it.

::: {.callout-important}
## Python 3.13 and the GIL
Starting with **Python 3.13** (October 2024), the GIL became optional. CPython now provides a "free-threaded" build mode in which the GIL is disabled, enabling full multi-threading for CPU-bound Python code. The standard build still includes the GIL for backwards compatibility.
:::

::: {.callout-warning}
## GIL Does Not Eliminate All Race Conditions
Even with the GIL, race conditions can occur during OS-level context switches. The GIL is released periodically — by default every 5 ms (`sys.getswitchinterval()`) — and briefly between the old thread releasing it and the new thread acquiring it. During that window, the OS scheduler may preempt execution mid-operation.

Compound operations that *appear* atomic in Python source code (e.g. `x += 1`, which compiles to LOAD/ADD/STORE bytecode instructions) are **not guaranteed to be atomic** under concurrent thread execution. A lock (`threading.Lock`) must still be used to protect shared state.

For a detailed analysis of the GIL's interaction with the OS scheduler, see: <https://www.dabeaz.com/python/GIL.pdf>
:::

## Threading vs Multiprocessing in Python

| | Threading (`threading`) | Multiprocessing (`multiprocessing`) |
|---|---|---|
| **Memory** | Shared across threads | Separate address space per process |
| **GIL** | Constrained by GIL (one thread executes at a time) | No GIL constraint (each process has its own interpreter) |
| **Best for** | I/O-bound tasks (disk, network) | CPU-bound tasks (numerical computation) |
| **Communication** | Shared variables (but need locks) | Queues, pipes, shared memory objects |

::: {.callout-tip}
Other libraries extend these capabilities: **`joblib`** provides a unified interface to both threading and multiprocessing backends.

```python
# joblib example: run square() in parallel across 4 processes
from joblib import Parallel, delayed

def square(x):
    return x ** 2

results = Parallel(n_jobs=4, backend='multiprocessing')(
    delayed(square)(i) for i in range(10)
)
# results = [0, 1, 4, 9, 16, 25, 36, 49, 64, 81]
```

**`mpi4py`** exposes the MPI (Message Passing Interface) standard to Python, enabling distributed-memory parallelism across cluster nodes — the same model used in large-scale HPC applications.
:::

Before the next two scripts, it is useful to define these terms precisely:

- **Synchronous (blocking)**: the calling code waits for the operation to complete before continuing. Execution is sequential.
