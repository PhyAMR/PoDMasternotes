9  SAP-3

The final step in the evolution towards modern computers includes three main features:

  1. Four new registers (D, E, H, L) that are added to the three already existing (A, B, C). They, in addition to giving greater freedom in programming, make programs faster, because - as already explained in the context of the SAP-2 - the instructions that act on the registers without going through the memory occupy a smaller number of timing states.

  2. The flags are indicated by a register F, and to the zero (Z) and sign (S) are added two more: the carry (CY), which acts as a carry or borrow (in other words, as the ninth bit of the accumulator), and the parity (P), active when the result of the ALU is composed of an even number of bits.

  3. The stack pointer, that is a \(16\)-bit register with the same role as the program counter dedicated to a specific area of memory called the stack, which we will talk about later.

9.1 Instructions

The SAP-3 inherits the entire instruction set of the SAP-2; between new registers and new instructions the total number goes from \(42\) to \(244\), although we will only see \(223\) of these.

They are detailed below, divided into appropriate categories, not necessarily coinciding with those of the SAP-2.

9.1.1 Data transfer instructions

All instructions are also present in the SAP-2; the only difference is that in MOV, for debugging reasons, the register from which the data is copied and the one in which it is passed can coincide.

9.1.2 Arithmetic instructions

The SAP-3 introduces the carry flag, indicated by CY, which allows greater freedom in arithmetic calculations by eliminating the possibility of overflow.

Two new instructions act exclusively on the carry flag:

  • (): activates the carry flag.

  • (): complements the carry flag.

To set the carry to 0 regardless of its current state, it is possible to first set the carry with STC and then complement it with CMC.

The sum and difference instructions (ADD X and SUB X) work in the same way as in the SAP-2, with the only difference that it is possible to include A as a register as well, which will thus add the content of the accumulator to itself.

To these are added two other similar instructions:

  • (): adds the content of the register X to the content of the accumulator, finally adding also the eventual carry.

    \[ \begin{cases} A=1000\hspace{0.8mm}0011\\ E=0001\hspace{0.8mm}0100\\ CY=1 \end{cases} \xrightarrow{\textit{ADC E}} \\ \begin{cases} A=1001\hspace{0.8mm}1000\\ E=0001\hspace{0.8mm}0100\\ CY=0 \end{cases} \]

  • (): subtracts the content of the register X from the content of the accumulator, finally removing also the eventual carry.

    \[ \begin{cases} A=1001\hspace{0.8mm}1000\\ L=1001\hspace{0.8mm}0110\\ CY=1 \end{cases} \xrightarrow{\textit{SBB L}} \\ \begin{cases} A=0000\hspace{0.8mm}0001\\ L=1001\hspace{0.8mm}0110\\ CY=0 \end{cases} \]

Another novelty in the SAP-3 are the immediate sums and differences:

  • (): adds the operand to the content of the accumulator.

    \[ A=0000\hspace{0.8mm}0001 \ \xrightarrow{\textit{ADI} \ 0\text{x}61} \ A=0110\hspace{0.8mm}0010 \]

  • (): subtracts the operand from the content of the accumulator.

    \[ A=0110\hspace{0.8mm}0010 \ \xrightarrow{\textit{SUI} \ 0\text{x}21} \ A=0100\hspace{0.8mm}0001 \]

  • (): adds the operand to the content of the accumulator, finally adding also the eventual carry.

    By activating the carry with STC we will have

    \[ A=0100\hspace{0.8mm}0001 \ \xrightarrow{\textit{ACI} \ 0\text{x}72} \ A=1011\hspace{0.8mm}0100 \]

  • (): subtracts the operand from the content of the accumulator, finally removing also the eventual carry.

    By activating the carry with STC we will have

    \[ A=1011\hspace{0.8mm}0100 \ \xrightarrow{\textit{SBI} \ 0\text{x}11} \ A=1010\hspace{0.8mm}0010 \]

Finally, the increment and decrement instructions (INR X and DCR X) remain the same as in the SAP-2.

9.1.3 Rotation instructions

The SAP-3 also inherits the RAL and RAR instructions, with the difference that the rotations also include the carry. Let’s see two examples of this:

\[ \begin{aligned} &\underbrace{\red{0}}_{\textit{CY}}\overbrace{\green{1}000\hspace{0.8mm}010\orange{1}}^{\textit{A}} \ \xrightarrow{\textit{RAL}} \ \underbrace{\green{1}}_{\textit{CY}}\overbrace{0000\hspace{0.8mm}10\orange{1}\red{0}}^{\textit{A}}\\ &\underbrace{\red{1}}_{\textit{CY}}\overbrace{\green{0}110\hspace{0.8mm}100\orange{0}}^{\textit{A}} \ \xrightarrow{\textit{RAR}} \ \underbrace{\orange{0}}_{\textit{CY}}\overbrace{\red{1}\green{0}11\hspace{0.8mm}0100}^{\textit{A}} \end{aligned} \]

We find the following results:

\[ \textit{RAL} \ \longrightarrow \\ \left\{ \begin{aligned} &\textit{CY}=\textit{MSB}\\ &\textit{LSB}=\textit{CY} \end{aligned} \right. \hspace{1.5cm} \textit{RAR} \ \longrightarrow \\ \left\{ \begin{aligned} &\textit{CY}=\textit{LSB}\\ &\textit{MSB}=\textit{CY} \end{aligned} \right. \]

If the carry is 0 before the rotation, the latter can be seen as a multiplication by two (in the case of RAL) or as an integer division by two (in the case of RAR). The reason is quite simple: by shifting all the bits one position to the left, all the contributions of the formula [eq:arithmetic-conversion-binary-to-decimal] will shift from \(2^i\) to \(2^{i+1}\), and this is equivalent to multiplying the entire summation, that is the number, by two; obviously the same will apply in the shift to the right, with the difference that the contributions will become \(2^{i-1}\), effectively dividing the number by two. It is important to underline that, for this to be valid, it is necessary that the carry flag is deactivated, since otherwise with the RAL instruction it would cause a contribution of \(2^0\) and with the RAR instruction a contribution of \(2^7\), both unwanted.

Two new instructions are the following:

  • (): stores the MSB in the carry flag and rotates the bits of the accumulator to the left.

    \[ \underbrace{\red{0}}_{\textit{CY}}\overbrace{\green{1}010\hspace{0.8mm}001\orange{0}}^{\textit{A}} \ \xrightarrow{\textit{RLC}} \ \underbrace{\green{1}}_{\textit{CY}}\overbrace{0100\hspace{0.8mm}01\orange{0}\green{1}}^{\textit{A}} \]

    \[ \textit{RLC} \ \longrightarrow \\ \begin{cases} \textit{CY}=\textit{MSB}\\ \textit{LSB}=\textit{MSB} \end{cases} \]

  • (): stores the LSB in the carry flag and rotates the bits of the accumulator to the right.

    \[ \underbrace{\red{1}}_{\textit{CY}}\overbrace{\green{0}100\hspace{0.8mm}010\orange{1}}^{\textit{A}} \ \xrightarrow{\textit{RLC}} \ \underbrace{\orange{1}}_{\textit{CY}}\overbrace{\orange{1}\green{0}10\hspace{0.8mm}0010}^{\textit{A}} \]

    \[ \textit{RRC} \ \longrightarrow \\ \begin{cases} \textit{CY}=\textit{LSB}\\ \textit{MSB}=\textit{LSB} \end{cases} \]

9.1.6 Extended-register instructions

The SAP-3 has the ability to also process \(16\)-bit data thanks to the extended registers. The six registers used to store bytes beyond the accumulator are in fact paired in the following way:

\[ \begin{aligned} &\textit{B} \ + \ \textit{C} \ \to \ \textit{\textbf{BC}}\\ &\textit{D} \ + \ \textit{E} \ \to \ \textit{\textbf{DE}}\\ &\textit{H} \ + \ \textit{L} \ \to \ \textit{\textbf{HL}} \end{aligned} \]

In the computer there is therefore a set of instructions that treats these pairs of registers as if they were single \(16\)-bit registers. Let’s see these instructions below:1

  • (): loads the indicated data into the extended register X.

  • (): adds the content of the extended register X to the content of HL.

    \[ \begin{cases} BC=1011\hspace{0.8mm}0000\hspace{0.8mm}0100\hspace{0.8mm}1100\\ HL=0000\hspace{0.8mm}0100\hspace{0.8mm}0010\hspace{0.8mm}1001 \end{cases} \xrightarrow{\textit{DAD B}} \\ \begin{cases} BC=1011\hspace{0.8mm}0000\hspace{0.8mm}0100\hspace{0.8mm}1100\\ HL=1011\hspace{0.8mm}0100\hspace{0.8mm}0111\hspace{0.8mm}1101 \end{cases} \]

  • (): adds \(0\)x\(0001\) to the content of the extended register X.

  • (): subtracts \(0\)x\(0001\) from the content of the extended register X.

9.1.7 Indirect instructions

With the instruction DAD X we have seen that the sum is performed between the extended register X and HL, and the result is saved precisely in HL. In fact, this register has a peculiarity that distinguishes it from the other two: it does not contain a data in itself, but a memory address in which a data is saved. In this sense we will say that it does not contain a data but points to it, so just as the program counter is an instruction pointer, the register HL will be a .

Let’s assume that the register HL contains \(0\)x\(2024\); if this register acts as a data pointer it means that it will point to the address \(R_{8228}\) and the read and write operations will refer to it. This address, in general, is indicated by M, and all the operations referred to it are called indirect since they do not concern it explicitly but refer to the pointer HL which acts as an intermediary for them.

9.1.8 Stack instructions

We had already introduced subroutines in the SAP-2 when talking about the CALL and RET instructions: they needed the last two memory addresses to save the point where the subroutine was called, so that with the return it was possible to return to the main program.

In the SAP-3 this concept is taken up and extended: an entire portion of memory2 - whose size is at the user’s discretion - is dedicated to the call and return addresses and to the status of the program. The pointer to the addresses of this memory is called the (indicated by SP) and is nothing more than a \(16\)-bit register analogous to the program counter.

Let’s see the first instruction useful for initializing the stack register:

  • : loads the indicated memory address into the stack register.

In this context, the stack counter is treated in all respects as an extended register, so the instructions DAD, INX and DCX will also be possible.

As mentioned, in the stack memory it is also possible to save the status of the program, that is the contents inside the registers so that they can be overwritten in the execution of the subroutines without being lost. Just as the memory addresses are saved in two addresses3 the stack instructions are made in such a way that with each of them the information is stored in two addresses: for this reason the extended registers are used, which contain precisely two bytes of information that can be stored in two addresses.

We know that there are three extended registers, namely BC, DE and HL. In this context, however, a fourth extended register is added, called PSW (Program Status Word), which contains the accumulator A and the flag register F; we remember, in fact, that although the flags in the SAP-3 are four, the register F is still an \(8\)-bit register,4 so in total PSW is actually made up of two bytes.

Let’s now see the two instructions that allow you to work with the program status:

  • : saves the content of the extended register X in the stack memory.

    In order, the operations performed are the following:

    1. The stack pointer is decremented, going from SP to SP-1

    2. The most significant byte is stored in the stack memory

    3. The stack pointer is decremented, going from SP-1 to SP-2

    4. The least significant byte is stored in the stack memory

    Assuming that the initialization of the stack pointer was given by LXI SP \(0\)x\(2000\), we will have

    \[ BC=0\text{x}A45B \ \xrightarrow{\textit{PUSH B}} \\ \begin{cases} R_{\num{8191}}=1010\hspace{0.8mm}0100\\ R_{\num{8190}}=0101\hspace{0.8mm}1011 \end{cases} \]

    being \(\num{8191}_{10}=0\text{x}1FFF\) and \(\num{8190}_{10}=0\text{x}1FFE\).

  • : copies the content from the stack memory into the extended register X.

    In order, the operations performed are the following:

    1. the content in the address indicated by SP is copied as the least significant byte into the extended register X

    2. The stack pointer is incremented, going from SP to SP+1

    3. the content in the address indicated by SP+1 is copied as the most significant byte into the extended register X

    4. The stack pointer is incremented, going from SP+1 to SP+2

    \[ \begin{cases} R_{\num{8190}}=0101\hspace{0.8mm}1011\\ R_{\num{8191}}=1010\hspace{0.8mm}0100 \end{cases} \xrightarrow{\textit{POP B}} \ \\ BC=0\text{x}A45B \]

The instructions PUSH and POP, therefore, are the inverse of each other. Let’s imagine the memory as a of addresses that are numbered from top to bottom: with the PUSH the stack pointer copies the data into memory (first the byte on the left, then the one on the right) going up; with the POP the stack pointer - going down - the data in the indicated register, copying first the byte on the right and then the one on the left.

In light of what has been said, it is easy to understand that the instructions CALL and RET in the SAP-3 have the role of PUSH and POP with respect to the program counter. The novelty is that of these instructions there are also the conditional versions, as for the jump instructions: we will therefore have CM () and CP () as well as RC () and RP () and so on.

Write a program that adds \(1200_{10}\) and \(3400_{10}\) and saves the result in the registers H* and L, both with and without the use of extended register instructions.*

First, let’s convert the two numbers to hexadecimal with the hex-dabble algorithm:

\[ \begin{aligned} 1200//16=75 \ & ext{with remainder} \ \red{0}\\ 75//16=4 \ & ext{with remainder} \ \red{11}\\ 4//16=0 \ & ext{with remainder} \ \red{4} \end{aligned} \]

\[ 1200_{10}=0\text{x}04B0 \]

\[ \begin{aligned} 3400//16=212 \ & ext{with remainder} \ \red{8}\\ 212//16=13 \ & ext{with remainder} \ \red{4}\\ 13//16=0 \ & ext{with remainder} \ \red{13} \end{aligned} \]

\[ 3400_{10}=0\text{x}0D48 \]

Let’s proceed first without the use of extended registers. In this case the registers are only able to store one byte, so we will first add the last two hexadecimal digits of the numbers in question, that is \(0\)x\(B0\) and \(0\)x\(48\). This sum will be stored in the register L which must contain the least significant byte.

Next, we will add the other two bytes, that is \(0\)x\(04\) and \(0\)x\(0D\), but also considering the eventual carry from the previous sum. This result will instead be saved in H, and at that point it will be possible to stop the program.

With the extended registers the program is greatly simplified: it will be enough to load \(0\)x\(04B0\) on an extended register and then put the other number \(0\)x\(0D48\) in HL; at that point, the DAD instruction will be sufficient to give us the desired result.

Using pointers, write a program that moves all the data contained between the memory addresses \(0\text{x}2030\) and \(0\text{x}212F\) to the memory addresses between \(0\text{x}5030\) and \(0\text{x}512F\).

We understand right away that we need a loop that takes us from the address \(0\)x\(2030\) to the address \(0\)x\(212F\), and in that same loop we must be able to also move between the new addresses in which to copy the data.

The steps to follow will be:

  1. We increment the register H, which points to the memory address from which to retrieve the data

  2. We copy the data into any register, for example the register B, and we copy the most significant byte of the address (that is \(0\)x\(20\)) into the accumulator A

  3. The difference between the old address and the new one is exactly \(0\)x\(30\) (always referring to the most significant byte), so we add this quantity to the accumulator

  4. We proceed in the opposite direction by copying the content of the accumulator into H and that of B into the new address to which H points

  5. To return to the addresses from which to retrieve the data, we subtract \(0\)x\(30\) from the accumulator and copy the result into H

  6. Now it’s time to figure out whether to proceed further or stop the cycle, so we copy the content of L into the accumulator and perform a comparison with \(0\)x\(2F\)

  7. If the difference is null then the address was the last one and we can stop the program, otherwise we will have to continue, so a JNZ associated with the address in which H is incremented at the first step will be enough to repeat the whole process

  1. In the mnemonics to refer to the extended register, only the first letter of the register is used: we will therefore have B for BC, D for DE and H for HL.↩︎

  2. Not necessarily located in the last memory addresses.↩︎

  3. We remember that a memory address is encoded by \(2\) bytes, and in it only one byte can be contained; in the SAP-2, in fact, two addresses were used to save the current point.↩︎

  4. The structure of this register depends on the specific model of SAP-3. In the 8085 microprocessor - one of the most widespread - it is constituted as follows: \(SZ00\hspace{0.8mm}0P0CY\). Beyond the specific position of the flags, the other four bits are always set to zero by default.↩︎

### Rotation instructions

The SAP-3 also inherits the *RAL* and *RAR* instructions, with the difference that the rotations also include the carry. Let's see two examples of this:

$$

$$
\textit{RAL} \ \longrightarrow \\
    \left\{
        \begin{aligned}
            &\textit{CY}=\textit{MSB}\\
            &\textit{LSB}=\textit{CY}
        \end{aligned}
    \right.
    \hspace{1.5cm}
    \textit{RAR} \ \longrightarrow \\

### Extended-register instructions

The SAP-3 has the ability to also process $16$-bit data thanks to the **extended registers**. The six registers used to store bytes beyond the accumulator are in fact paired in the following way:

$$
\begin{aligned}
    &\textit{B} \ + \ \textit{C} \ \to \ \textit{\textbf{BC}}\\
    &\textit{D} \ + \ \textit{E} \ \to \ \textit{\textbf{DE}}\\
    &\textit{H} \ + \ \textit{L} \ \to \ \textit{\textbf{HL}}
\end{aligned}
$$

In the computer there is therefore a set of instructions that treats these pairs of registers as if they were single $16$-bit registers. Let's see these instructions below:[^1]

-   (): loads the indicated data into the extended register *X*.

-   (): adds the content of the extended register *X* to the content of *HL*.

                HL=0000\hspace{0.8mm}0100\hspace{0.8mm}0010\hspace{0.8mm}1001

$$

-   (): adds $0$x$0001$ to the content of the extended register *X*.

-   (): subtracts $0$x$0001$ from the content of the extended register *X*.

### Indirect instructions

With the instruction *DAD X* we have seen that the sum is performed between the extended register *X* and *HL*, and the result is saved precisely in *HL*. In fact, this register has a peculiarity that distinguishes it from the other two: it does not contain a data in itself, but a *memory address* in which a data is saved. In this sense we will say that it does not contain a data but *points* to it, so just as the program counter is an *instruction pointer*, the register *HL* will be a .

Let's assume that the register *HL* contains $0$x$2024$; if this register acts as a data pointer it means that it will point to the address $R_{8228}$ and the read and write operations will refer to it. This address, in general, is indicated by *M*, and all the operations referred to it are called **indirect** since they do not concern it explicitly but refer to the pointer *HL* which acts as an intermediary for them.

### Stack instructions

We had already introduced subroutines in the SAP-2 when talking about the *CALL* and *RET* instructions: they needed the last two memory addresses to save the point where the subroutine was called, so that with the return it was possible to return to the main program.

In the SAP-3 this concept is taken up and extended: an entire portion of memory[^2] - whose size is at the user's discretion - is dedicated to the call and return addresses and to the *status* of the program. The pointer to the addresses of this memory is called the (indicated by *SP*) and is nothing more than a $16$-bit register analogous to the program counter.

Let's see the first instruction useful for initializing the stack register:

-   : loads the indicated memory address into the stack register.

In this context, the stack counter is treated in all respects as an extended register, so the instructions *DAD*, *INX* and *DCX* will also be possible.

As mentioned, in the stack memory it is also possible to save the *status* of the program, that is the contents inside the registers so that they can be overwritten in the execution of the subroutines without being lost. Just as the memory addresses are saved in two addresses[^3] the stack instructions are made in such a way that with each of them the information is stored in two addresses: for this reason the extended registers are used, which contain precisely two bytes of information that can be stored in two addresses.

We know that there are three extended registers, namely *BC*, *DE* and *HL*. In this context, however, a fourth extended register is added, called **PSW** (*Program Status Word*), which contains the accumulator *A* and the flag register *F*; we remember, in fact, that although the flags in the SAP-3 are four, the register *F* is still an $8$-bit register,[^4] so in total *PSW* is actually made up of two bytes.

Let's now see the two instructions that allow you to work with the program status:

-   : saves the content of the extended register *X* in the stack memory.

    In order, the operations performed are the following:

    1.  The stack pointer is decremented, going from *SP* to *SP-1*

    2.  The most significant byte is stored in the stack memory

    3.  The stack pointer is decremented, going from *SP-1* to *SP-2*

    4.  The least significant byte is stored in the stack memory

    Assuming that the initialization of the stack pointer was given by *LXI SP* $0$x$2000$, we will have

    $$
BC=0\text{x}A45B \ \xrightarrow{\textit{PUSH B}} \\
            \begin{cases}
                R_{\num{8191}}=1010\hspace{0.8mm}0100\\
                R_{\num{8190}}=0101\hspace{0.8mm}1011
            \end{cases}
$$

    being $\num{8191}_{10}=0\text{x}1FFF$ and $\num{8190}_{10}=0\text{x}1FFE$.

-   : copies the content from the stack memory into the extended register *X*.

    In order, the operations performed are the following:

    1.  the content in the address indicated by *SP* is copied as the least significant byte into the extended register *X*

    2.  The stack pointer is incremented, going from *SP* to *SP+1*

    3.  the content in the address indicated by *SP+1* is copied as the most significant byte into the extended register *X*

    4.  The stack pointer is incremented, going from *SP+1* to *SP+2*

    $$
\begin{cases}
                R_{\num{8190}}=0101\hspace{0.8mm}1011\\
                R_{\num{8191}}=1010\hspace{0.8mm}0100
            \end{cases}
            \xrightarrow{\textit{POP B}} \ \\
            BC=0\text{x}A45B
$$

The instructions *PUSH* and *POP*, therefore, are the inverse of each other. Let's imagine the memory as a of addresses that are numbered from top to bottom: with the *PUSH* the stack pointer copies the data into memory (first the byte on the left, then the one on the right) going up; with the *POP* the stack pointer - going down - the data in the indicated register, copying first the byte on the right and then the one on the left.

In light of what has been said, it is easy to understand that the instructions *CALL* and *RET* in the SAP-3 have the role of *PUSH* and *POP* with respect to the program counter. The novelty is that of these instructions there are also the *conditional* versions, as for the jump instructions: we will therefore have *CM* () and *CP* () as well as *RC* () and *RP* () and so on.

Let's proceed first without the use of extended registers. In this case the registers are only able to store one byte, so we will first add the last two hexadecimal digits of the numbers in question, that is $0$x$B0$ and $0$x$48$. This sum will be stored in the register *L* which must contain the least significant byte.

Next, we will add the other two bytes, that is $0$x$04$ and $0$x$0D$, but also considering the eventual carry from the previous sum. This result will instead be saved in *H*, and at that point it will be possible to stop the program.

With the extended registers the program is greatly simplified: it will be enough to load $0$x$04B0$ on an extended register and then put the other number $0$x$0D48$ in *HL*; at that point, the *DAD* instruction will be sufficient to give us the desired result.

::: {.example}
*Using pointers, write a program that moves all the data contained between the memory addresses $0\text{x}2030$ and $0\text{x}212F$ to the memory addresses between $0\text{x}5030$ and $0\text{x}512F$.*

We understand right away that we need a loop that takes us from the address $0$x$2030$ to the address $0$x$212F$, and in that same loop we must be able to also move between the new addresses in which to copy the data.

The steps to follow will be:

1.  We increment the register *H*, which points to the memory address from which to retrieve the data

2.  We copy the data into any register, for example the register *B*, and we copy the most significant byte of the address (that is $0$x$20$) into the accumulator *A*

3.  The difference between the old address and the new one is exactly $0$x$30$ (always referring to the most significant byte), so we add this quantity to the accumulator

4.  We proceed in the opposite direction by copying the content of the accumulator into *H* and that of *B* into the new address to which *H* points

5.  To return to the addresses from which to retrieve the data, we subtract $0$x$30$ from the accumulator and copy the result into *H*

6.  Now it's time to figure out whether to proceed further or stop the cycle, so we copy the content of *L* into the accumulator and perform a comparison with $0$x$2F$

7.  If the difference is null then the address was the last one and we can stop the program, otherwise we will have to continue, so a *JNZ* associated with the address in which *H* is incremented at the first step will be enough to repeat the whole process
:::
